# 图谱微调操作规范（评测阶段 + 微调阶段）

本文档与下列实现对齐：`evaluation.py`、`refinement.py`、`recluster.py`、`pipeline.py`，以及 `prompts/Evaluation_Prompt.txt`、`Cluster_Evaluation_Prompt.txt`、`Refinement_Plan_Prompt.txt`。

---

## 一、职责划分总览

| 来源 | 允许的 `adjustment_suggestions` | 关键字段（id 与 name 成对） |
|------|------------------------------|---------------------------|
| **全局评测** | 仅 `DELETE`（批量孤立）+ `ADD`（向某 **L1** 补 **L2**） | `DELETE`：`delete_mode=isolated_batch`；`ADD`：`parent_id` + **`parent_name`**（与 L1 目录一致） |
| **L1 簇评测** | `ADD`（向 **L2** 补 **L3**）、`MERGE`（**L2 或 L3** 两点）、`DELETE`（单点） | `ADD`：`parent_id` + `parent_name`；`MERGE`：`keep_id`/`keep_name`、`remove_id`/`remove_name`；`DELETE`：`target_id` + `target_name` |

**不再实现**：L1 下 L2>10 时由微调自动「压栈」合并（无 `execute_merge_l1` / 贪心合并）。评测仍可 **不通过** 并写入说明。

**硬约束**：每个 L1 下 L2 ≤ 10；向 L1 执行 ADD（补 L2）时受剩余槽位与 `max_add_nodes_per_action` 限制。

---

## 二、评测阶段

**产出**：`evaluation_report.adjustment_suggestions`（全局 + 各簇合并；簇项带 `source: L1簇:<L1名称>`）。

**后处理**（`evaluation._attach_ids_to_suggestions`）：按图补全缺失的 id（名称唯一时）；并按节点 **自动补齐** `parent_name` / `keep_name` / `remove_name` / `target_name`（模型漏写时）。

**落盘**（`_save_evaluation_artifacts`）：每次评测写入 `data/output/evaluation/<run_id>/`（含各簇 `clusters/*.md`），并同步 `data/output/final_evaluation/`。

### 2.1 L1 簇上下文

簇评测中 `{l2_l3_tree}` 为 **JSON 树**：根 `l1`，`children` 为 L2 列表，每项含 `children`（L3）。模型须从该 JSON **逐字复制 id**；**name 字段**须与 JSON 中对应 `name` 一致。

### 2.2 全局评测仅允许

- **DELETE**：`delete_mode=isolated_batch`，`target_id` 可空或占位。
- **ADD**：`parent_id` + `parent_name`，与 `l1_id_catalog` 一致。

### 2.3 L1 簇评测仅允许

- **ADD**：`parent_id` + `parent_name`（L2）。
- **MERGE**：`keep_id`/`keep_name`、`remove_id`/`remove_name`。
- **DELETE**：`target_id` + `target_name`（单点）。

**禁止**：簇内输出「删除全部孤立节点」。

---

## 三、校准后 L2 重聚合（`recluster`）

对每个 L1：生成新聚合 L2；**原 L2 下沉为 L3、原 L3 下沉为 L4**；随后将下沉节点的 **id 前缀规范为 `l3_` / `l4_`**，并全图重写引用（`recluster._apply_sink_id_renames`）。

---

## 四、微调阶段

### 4.1 输入

`evaluation_report`、`calibrated_kps` / `calibrated_relationships`。

### 4.2 微调计划 LLM（`generate_refinement_plan`）

**提示词注入**：

- `high_level_suggestions`：多行文本，**每条建议 id 与 name 同框**（优先评测输出的 `*_name`，并与当前图核对）。
- `l1_catalog_json`：全量 L1（全局 ADD 父节点）。
- **`suggestion_nodes_catalog_json`**：仅本批建议中出现的节点（**不全量 L2**），含 `id` / `name` / `level` / 父边信息。
- `sparse_json`、`isolated_json`、`overloaded_json`（过载仅信息，禁止据此编造 L1 级 MERGE）、`max_actions`。

**输出**：`{"actions": [...]}`，经解析过滤后 `_sort_actions`：**MERGE → ADD → DELETE**。计划 LLM 须在每条 action 中 **id 与 name 成对**（与目录/建议一致）；执行以 id 为准。

| `type` | 必填 | 说明 |
|--------|------|------|
| `ADD` | `parent_id`+`parent_name`，可选 `focus` | 父为 L1 → 补 L2；父为 L2 → 补 L3 |
| `MERGE` | `keep_id`/`keep_name`、`remove_id`/`remove_name` 或 `target_id`+`source_id` | 仅两点合并 |
| `DELETE` | `isolated_batch` 或 `single` + `target_id`+`target_name` | 批量时 name 可空 |

**ADD 执行侧 LLM**（`execute_add`）：生成新节点时须遵守 **`l2_`/`l3_` + 英文 snake_case** 的 id 规则与**规范中文知识点 name**，并与同级已有节点的 id/name 风格对齐（提示词内注入同级示例）。  
对 **L3** 另有提示词约束：**粒度与去重**（避免过细拆分、与已有 L3 同义重复）；数量受 `refinement.max_l3_per_l2`（默认单 L2 下 L3 总数上限）、`refinement.max_l3_add_per_call`（默认单次补充条数上限）与剩余槽位共同限制。

### 4.3 执行（`GraphRefinementAgent.execute`）

`MERGE` → `merge_two_kp_ids`；`ADD` → `execute_add`；`DELETE` → `execute_delete`。写回 `state` 与 `data/output/calibrated_*.parquet`。

---

## 五、评测 → 计划 → 执行映射

| 评测 action + 来源 | 评测必填 | 计划 `type` | 执行 |
|--------------------|----------|-------------|------|
| DELETE 全局 | `delete_mode=isolated_batch` | `DELETE` 批量 | 删全部孤立节点 |
| ADD 全局 | `parent_id`（+ `parent_name`） | `ADD` | 生成 L2 |
| ADD 簇 | `parent_id`（+ `parent_name`） | `ADD` | 生成 L3 |
| MERGE 簇 | `keep_id`/`remove_id`（+ name） | `MERGE` | 两点合并 |
| DELETE 簇 | `target_id`（+ `target_name`） | `DELETE` single | 删该节点及边 |

---

## 六、流水线评估循环（`pipeline.run_full_pipeline`）

- `--max-eval-loops`：评测轮数上限。
- **未通过**时：在最后一轮之前执行 `tune_graph`；若 **`max_eval_loops == 1`**，在第一轮评测后 **仍会执行一次微调**（再评测需将 `--max-eval-loops` 设为 ≥2）。
 - 默认 `--max-eval-loops`：**5**。

修订其它提示词或字段时，请同步核对本节与上文表格。
