# 全流程细节级实现说明（代码对齐版）

本文档按“可落地排查”的粒度描述当前实现：  
**函数入口 -> 输入 -> 处理细节 -> 状态变更 -> 落盘文件 -> 异常分支 -> 日志定位点**。

对应代码：
- 主流程：`knowledge_graph/pipeline.py`
- CLI入口：`knowledge_graph/__main__.py`
- 各步骤实现：`knowledge_graph/agents/*.py`

---

## 1. 运行入口与参数

### 1.1 CLI入口

命令入口是 `python -m knowledge_graph`，在 `knowledge_graph/__main__.py` 中解析参数：

- `step`：默认 `full`，也可单步执行（`extract_l1` / `validate_l1` / `extract_l1_rels` / `extract` / `vectorize` / `calibrate` / `evaluate` / `build`）。
- `--test`：测试模式，限制章节数。
- `--max-loops`：步骤1-2（L1提取与验证）最大循环次数。
- `--max-eval-loops`：步骤7-7.5（评测-微调）最大循环次数。
- `--incremental`：增量模式（依赖 `data/processed_files.json`）。
- `--full-refresh-l1`：强制重跑步骤1-2（否则在满足跳过条件时可能复用 `stage1_entities.parquet`）。

等价全流程入口：`python -m knowledge_graph.pipeline`，参数与上述 **`full` 相关开关**一致（`--max-loops` / `--max-eval-loops` / `--test` / `--incremental` / `--full-refresh-l1`），区别是不使用 `step` 子命令。

### 1.2 主函数

`run_full_pipeline(config, max_loops, max_eval_loops, test_mode, incremental, skip_l1_if_exists, ...)`：

1. 初始化 `PipelineState`（包含所有跨步骤共享字段）。
2. 阶段一：执行 `extract_l1 -> validate_l1`，并依据 `validation_errors` 决定是否回环。
3. 阶段二：执行 `extract_prerequisites -> extract_entities -> vectorize -> calibrate -> regroup_graph`。
4. 阶段三：执行 `evaluate`，不通过则进 `tune_graph`，再回评测，直到通过或达上限。
5. 最终执行 `build_graph` 写入 Neo4j。

---

## 2. 全局状态字段（PipelineState）关键项

以下字段是跨阶段最关键的“事实源”：

- `validated_l1_concepts`：L1最终候选（步骤2后可信）。
- `l1_prerequisites`：L1前置关系（步骤3）。
- `knowledge_points/resources/relationships`：步骤4产物（未校准）。
- `calibrated_kps/calibrated_resources/calibrated_relationships`：步骤6后统一主数据（步骤6.5与7.5会继续更新）。
- `evaluation_report`：步骤7主评测结果（含 `adjustment_suggestions`）。
- `tuning_summary`：步骤7.5动作执行统计与是否生效。
- `recluster_applied`：步骤6.5单次执行保护位。
- `errors`：任何步骤错误都会追加到此列表。

## 3. 步骤1：L1提取（`L1ExtractorAgent.execute`）

### 3.1 输入

- `data/input/Table_of_Contents/*.txt` 目录文本。
- `config.pipeline.subject`。
- 可选反馈：上轮 `validation_errors` 转换为 `feedback`。

### 3.2 处理细节

1. `load_toc_files`：遍历并读取全部 TOC 文件。
2. `_extract_chapters`：使用正则提取章节标题。
3. `format_chapters_for_prompt`：组装章节列表。
4. `load_prompt`：默认 `prompts/L1_Extraction_Prompt.txt`；若有反馈且文件存在，切换 `L1_Extraction_With_Feedback_Prompt.txt`。
5. `call_llm`：请求 LLM 生成 L1 列表。
6. 对响应做“括号兜底”修复（补 `[`/`]`），再 `json.loads`。
7. 生成标准 L1 实体字段：`id=l1-{i}`, `name`, `definition`, `level=L1`, `source_chapters`。

### 3.3 状态变更与落盘

- `state.l1_concepts = l1_points`
- `state.l1_extraction_prompt = prompt`
- `state.current_step = extract_l1`
- 落盘：`data/output/stage1_entities.parquet`

### 3.4 异常分支

- JSON解析失败：`state.errors += 解析LLM响应失败`
- 其他异常：`state.errors += str(e)`

---

## 4. 步骤2：L1验证（`L1ValidatorAgent.execute`）

### 4.1 输入

- 优先 `state.l1_concepts`，为空则退回 `state.validated_l1_concepts`。

### 4.2 处理细节

1. `validate_holistic`：将全部 L1 组装成列表文本，调用 `prompts/L1_Validation_Prompt.txt`。
2. 返回结构核心字段：`is_valid`, `overall_score`, `dimensions`, `overall_feedback`, `issues_found`。
3. 给每个 L1 注入同一份整体验证结果（当前实现是“整体评分挂到每个概念”）。
4. 若 `is_valid=false`，构造 `validation_errors=[{name:'整体系统', feedback, total_score}]`。

### 4.3 状态变更与落盘

- `state.validated_l1_concepts`
- `state.validation_summary`
- `state.validation_errors`
- `state.current_step = validate_l1`
- 落盘：`data/output/stage1_entities.parquet`（覆盖为验证版）

### 4.4 循环判定（在 pipeline 中）

- `validation_errors` 非空且未达 `max_loops`：回到步骤1。
- 否则继续步骤3。

---

## 5. 步骤3：L1前置关系（`L1PrerequisiteAgent.execute`）

### 5.1 输入

- `state.validated_l1_concepts`

### 5.2 处理细节

1. 提取 L1 名称列表，注入 `prompts/L1_Prerequisite_Prompt.txt`。
2. `call_llm` 生成 `prerequisites`。
3. 将 `from_l1/to_l1` 名称映射回真实 L1 ID。
4. 仅在映射成功时写关系：
   - `type=prerequisite`
   - `start_id`/`end_id`
   - `end_type=L1`
   - `reason`

### 5.3 状态变更与落盘

- `state.l1_prerequisites`
- `state.current_step = extract_prerequisites`
- 落盘：`data/output/stage2_relationships.parquet`

---

## 6. 步骤4：实体与关系抽取（`EntityExtractorAgent.execute`）

### 6.1 输入

- `state.validated_l1_concepts`
- `data/input/*.csv`（排除目录文件）
- `data/processed_files.json`（增量处理记录）

### 6.2 处理细节

1. `load_textbook_data`：
   - 扫描所有 CSV；
   - 与 `processed_files` 对比，仅标记新增文件行（`is_new=true`）。
2. 测试模式下切片：只保留前 `max_chapters`。
3. 若 `new_chunks` 为空：直接回读历史 `stage3` 三份 parquet 并返回（跳过 LLM）。
4. 对每个新增 chunk：
   - 组装资源链接字典（lecture/ppt/code/video）；
   - `extract_from_chunk` 调 LLM，文本截断到 5000 字；
   - 读取 `l2_points/l3_points/relationships/resources`。
5. 统一补层级：
   - `l2_points -> level=L2`
   - `l3_points -> level=L3`
6. 关系端点修正：
   - 若关系 `start_id` 实际是 L1 名称，则映射为 L1 真 ID。
7. 与历史 stage3 结果合并去重：
   - 实体按 `id` 去重；
   - 关系按 `(type,start_id,end_id)` 去重；
   - 资源按 `url` 去重。
8. 更新 `processed_files.json`。

### 6.3 状态变更与落盘

- `state.knowledge_points/relationships/resources`
- `state.current_step = extract_entities`
- 落盘：
  - `data/output/stage3_entities.parquet`
  - `data/output/stage3_relationships.parquet`
  - `data/output/stage3_resources.parquet`

---

## 7. 步骤5：向量化（`VectorizationAgent.execute`）

### 7.1 输入

- `state.knowledge_points + state.resources`

### 7.2 处理细节

1. 初始化 `VectorDBManager(config, force_recreate=True)`。
2. `add_entities(all_entities)` 批量写入向量库。

### 7.3 状态变更

- `state.vector_db_updated = True`
- `state.current_step = vectorize`

### 7.4 异常分支

- 若无可向量化实体：写 warning 并追加 `state.errors`。

## 8. 步骤6：数据校准（`CalibrationAgent.execute`）

### 8.1 输入

- L1：`state.validated_l1_concepts`
- stage2：`stage2_relationships.parquet`
- stage3：`stage3_entities/relationships/resources.parquet`

### 8.2 处理链路（核心）

1. `merge_all_entities`：合并 L1 + stage3 实体。
2. `deduplicate`（精确去重）：
   - 知识点按 `name.lower().strip()`
   - 资源按 `url`
3. `load_all_relationships`：合并 state 和 parquet 的关系来源。
4. `deduplicate_with_similarity`：
   - 先字符串相似度筛候选（阈值 `0.8`）；
   - 再调用 LLM 判定 `should_merge + confidence>=0.7`；
   - 执行实体合并和关系 ID 替换。
5. `assign_hierarchy`：
   - 根据 contains 关系回填 `parent_id/parent_name`。
6. `validate_and_calibrate_relationships`（最关键）：
   - 去重关系；
   - 校验端点存在与层级合法；
   - contains 仅允许：
     - `L1->L2`
     - `L2->L3`
     - `L3->L4`
   - prerequisite 仅允许同层；
   - 限制 L2/L3 多父；
   - 检查反向同类边（cycle）并移除。
7. `merge_definition_description`：
   - 若同时存在 `definition` 和 `description`，合并后删除 `description`。

### 8.3 状态变更与落盘

- `state.calibrated_kps/resources/relationships`
- `state.current_step = calibrate`
- 落盘：
  - `data/output/calibrated_entities.parquet`
  - `data/output/calibrated_resources.parquet`
  - `data/output/calibrated_relationships.parquet`

---

## 9. 步骤6.5：L2重聚合（`GraphReclusterAgent.execute`）

### 9.1 执行保护

- 若 `state.recluster_applied=true`，直接跳过，防止重复下沉。

### 9.2 输入

- 最新 `calibrated` 三份数据（优先 state，兜底 parquet）。
- `recluster.max_l2_per_l1`（默认10）。
- `prompts/Recluster_Prompt.txt`。

### 9.3 单个L1簇处理（`_apply_one_l1_recluster`）

1. 找出 `L1 -> old L2`（contains）集合。
2. 构造 LLM 聚合请求：
   - L1信息；
   - old_l2 节点列表（id/name/definition/aliases）。
3. 解析结果（`_parse_cluster_result`）并强校验：
   - `new_l2_nodes` 合法且截断至上限；
   - `mapping` 只接受合法 `new_l2_id`；
   - 强制 `old_l2_id` 单归属（owner表）；
   - 未覆盖 old_l2 自动补挂到默认簇；
   - 新L2前置关系仅保留合法边。
4. 层级下沉：
   - old L2 改为 L3；
   - old L3 改为 L4。
5. 关系重写：
   - 删除旧 `L1->oldL2 contains`；
   - 新增 `L1->newL2 contains`；
   - 新增 `newL2->oldL2(now L3) contains`；
   - 追加新L2间 prerequisite；
   - 关系去重。
6. 回填 oldL2 的 `parent_id/parent_name` 到新L2。

### 9.4 故障兜底

- LLM返回非法时走 `_fallback_clusters`（均匀分桶聚合）。
- 单个L1异常会回滚该簇（不影响其他簇继续）。

### 9.5 状态变更与落盘

- 更新 `state.calibrated_kps/relationships/resources`
- `state.current_step = recluster`
- `state.recluster_applied = True`
- 落盘：
  - `data/output/calibrated_entities.parquet`
  - `data/output/calibrated_relationships.parquet`

---

## 10. 步骤7：评测（`EvaluationAgent.execute`）

### 10.1 输入

- 最新 `state.calibrated_kps/resources/relationships`
- 评测提示词：
  - `prompts/Evaluation_Prompt.txt`
  - `prompts/Cluster_Evaluation_Prompt.txt`
- 课程信息：`config.pipeline.subject`

### 10.2 评测数据准备（`prepare_evaluation_data`）

1. 统计全图数据：
   - 节点数、关系数、L1/L2/L3数量、孤立节点等。
2. 计算结构指标：
   - 连通分量、度分布、层级分布、关系类型分布。
3. 计算层级质量：
   - L2/L3有父率、平均每层子节点数。
4. 计算关系质量：
   - prerequisite/contains/has_resource 占比。
5. 计算源覆盖启发信息（稀疏主题关键词命中）。

### 10.3 LLM评测执行

1. 先跑全局评测，解析 JSON（失败时触发 JSON 修复二次请求）。
2. 再对每个 L1 跑簇评测：
   - 构造 L2/L3 树；
   - 收集 `issues` 与 `adjustment_suggestions`；
   - 建立 `cluster_details`。
3. 聚合建议后，执行硬门槛：
   - `_collect_over_limit_l1_topics` 找 `L2>10` 主题；
   - `_enforce_l2_cap_gate` 强制 `is_passed=false` 并补充 `MERGE` 高优先级建议。
4. 通过判定：
   - 结果显式 `is_passed` 优先；
   - 旧结构缺失时按 `score>7.5 且 blocking_issues为空` 推断。

### 10.4 状态变更与落盘

- `state.evaluation_report`
- `state.evaluation_passed`
- `state.current_step = evaluate`
- 分轮归档（每轮评测一个 `run_id` 目录）：
  - `data/output/evaluation/<run_id>/bundle.json`
  - `data/output/evaluation/<run_id>/overall_result.json`
  - `data/output/evaluation/<run_id>/prepared_data.json`
  - `data/output/evaluation/<run_id>/cluster_results.json`
  - `data/output/evaluation/<run_id>/evaluation_report.md`
  - `data/output/evaluation/<run_id>/clusters/*.md`
- 历史快照：
  - `data/output/evaluation/latest.json`
- 最新评估同步输出：
  - `data/output/final_evaluation/**`（含 `clusters/*.md`）

## 11. 步骤7.5：微调（`GraphRefinementAgent.execute`）

### 11.1 输入

- `state.evaluation_report.adjustment_suggestions`
- 最新 `calibrated` 图数据（优先 state）。

### 11.2 执行前预处理

1. `load_current_graph` 取最新图。
2. 先做一次 `validate_relationships`（复用校准关系校验）：
   - 目的是清洗历史脏关系，避免动作后被“固定脏数据变化”干扰回滚判定。
3. 计算当前稀疏 L1 集合（`L2<10`）。

### 11.3 计划生成（`generate_refinement_plan`）

1. 输入裁剪：
   - 仅保留 `priority=high`；
   - 最多 `max_input_suggestions`（代码默认 60，可配置为 `refinement.max_input_suggestions`）。
2. ADD 预归一：
   - 优先 `target_id` 命中稀疏L1；
   - 再 `target(name)` 命中；
   - 再从 `source` 的 `L1簇:xxx` 推断；
   - 无法归一则直接丢弃。
3. 计划提示词约束：
   - 动作数最多 `max_actions`（代码默认 35，可配置为 `refinement.max_actions`）。
4. 输出解析与过滤：
   - 支持 `MERGE` / `ADD` / `DELETE`（并非仅 ADD）；
   - 动作会按 `_ACTION_ORDER` 排序：**MERGE → ADD → DELETE**；
   - 对动作中涉及的 id 做解析/对齐（尽量落到图中真实 id），无法解析的动作会跳过。

### 11.4 ADD执行（`execute_add`）

1. 父节点解析优先级：
   - `target_id`
   - `source` 推断
   - `target` 名称
2. 强约束（以代码为准）：
   - 父节点必须存在，且 level 为 `L1` 或 `L2`；
   - 若父为 `L1`：补 `L2`，且 `L1` 下 `L2` 不得超过 10；
   - 若父为 `L2`：补 `L3`，受 `refinement.max_l3_per_l2`、`refinement.max_l3_add_per_call` 等上限约束；
   - 单次新增还受 `refinement.max_add_nodes_per_action` 限制，并结合剩余槽位动态裁剪。
3. LLM返回处理：
   - 仅接受 `new_nodes` 列表；
   - 每个节点字段归一（`_normalize_added_node`）；
   - ID 非法或冲突则 fallback 生成；
   - `difficulty` 不在白名单回退“中等”；
   - 同一父节点下名称去重。
4. 落图：
   - 新增 L2 节点（补 `parent_id/parent_name`）；
   - 新增 `contains(L1->L2)`（防重复）。

### 11.5 动作级回滚（关键）

每个动作执行后，比较前后：

- `kp_count`
- `rel_count`
- `isolated_count`

但回滚判定只看“**本次新增节点是否变孤立**”：
- `added_ids = after_kp_ids - before_kp_ids`
- 若 `added_ids` 与 `after_isolated_ids` 有交集，则回滚该动作。

### 11.6 状态变更与落盘

- 更新 `state.calibrated_kps/relationships`（资源不入图；此处不处理资源）
- 落盘：
  - `data/output/calibrated_entities.parquet`
  - `data/output/calibrated_relationships.parquet`
- `state.tuning_summary`：
  - `applied`
  - `actions_count`
  - `applied_actions_count`
  - `new_kps/new_relationships`

## 12. 步骤8：Neo4j构建（`GraphBuilderAgent.execute`）

### 12.1 输入

- `state.calibrated_kps/relationships`
- Neo4j连接参数：`config.neo4j`

### 12.2 处理细节

1. 建立 driver 与 session。
2. 执行 `MATCH (n) DETACH DELETE n` 清空库（全量覆盖策略）。
3. 按层级分批创建 `KnowledgePoint`：
   - L1（id/name/definition/level）
   - L2（额外写 `parent_id`）
   - L3
   - L4（已支持）
4. 创建关系：
   - `contains`
   - `prerequisite`

> 当前实现会过滤 `has_resource`，不写入资源与资源边（见 `knowledge_graph/agents/graph_builder.py`）。

### 12.3 状态变更

- `state.neo4j_imported = True`
- `state.current_step = build_graph`

## 13. 评测-微调循环的终止条件（精确）

在 `pipeline.py` 中，`for eval_iteration in range(1, max_eval_loops+1)`：

1. 若 `evaluate` 报错 -> 终止循环。
2. 若 `report.is_passed=true` -> 终止循环（通过）。
3. 若达到 `max_eval_loops` -> 终止循环（未通过也停止）。
4. 否则执行 `tune_graph`。
5. 若 `tuning_summary.applied=false` -> 终止循环（无可执行动作）。
6. 否则进入下一轮评测。

## 14. 关键日志定位（排障必看）

### 14.1 主流程日志关键词

- `=== 迭代 x/y ===`：L1验证循环轮次
- `=== 评估迭代 x/y ===`：评测微调轮次
- `评测结果: score=..., passed=...`

### 14.2 重聚合日志关键词

- `重聚合[xxx]: 原L2=... -> 新L2=...`
- `重聚合失败，回滚该L1簇`

### 14.3 微调日志关键词

- `微调计划: N 个操作`
- `动作效果[ADD]: kp a->b, rel c->d, isolated e->f`
- `动作回滚[ADD]: 本次新增节点出现孤立`
- `ADD跳过: 目标L1非稀疏主题`
- `ADD跳过: ... 下L2已达上限10`

## 15. 你可以如何用这份文档做代码验收

建议按以下清单逐项验收：

1. 跑一轮 `full --max-eval-loops 1`，验证“无微调时”的评测输出完整性。
2. 跑一轮 `full --max-eval-loops 2`，验证“至少一次微调动作”的有效性。
